# Техническое видение: forum-etl

## Продукт и цель

Сервис преобразует экспорт топиков форума (файлы в каталоге `data/`) в структурированные записи, пригодные для дальнейшего построения эмбеддингов и загрузки в **Qdrant** для сценариев **RAG**.

На **первом этапе** целевой результат — не векторы, а **нормализованные объекты** (JSON по согласованной схеме): идентификаторы, текст, цитаты и связи между постами. Поля, предназначенные для текста под эмбеддинг, на этом этапе заполняются частично (см. ниже).

## Ограничения и допущения

- Один файл в `data/` соответствует **одному топику** форума.
- Исходный формат и правила парсинга считаются **контрактом экспорта**; при смене формата выгрузки правила в этом документе и в коде должны быть явно обновлены.
- Цитаты для связей и метаданных учитываются только с **`depth: 1`** в структуре `quotes` сообщения; вложенные уровни (если появятся) в текущей постановке не используются.
- Сопоставление текста цитаты с предыдущими постами — **эвристика** (RapidFuzz); гарантия «как в БД форума» не требуется, требуется предсказуемое и воспроизводимое поведение по правилам ниже.
- **Минимальной длины цитаты нет:** короткие строки обрабатываются теми же правилами (риск ложных совпадений осознанно принимается).
- Язык реализации: **Python**. Зависимости для текущего этапа: **PyYAML** (разбор документов), **RapidFuzz** (сопоставление цитат). Зависимости для последующих этапов (клиент Qdrant, модель эмбеддингов) вводятся отдельно, по мере задачи.

## Структура каталога `data/`

- Именование файлов: `topic_<topic_id>.txt` (пример: `topic_2036930.txt`).
- Содержимое файла — **последовательность YAML-документов**, разделённых строкой, содержащей только **`---`**, в типичном для YAML виде «мульти-документ».
- Первый документ в файле: **`kind: topic`** — метаданные топика (идентификаторы форума и топика, заголовок, счётчики, URL и т. п.).
- Последующие документы: **`kind: message`** — отдельные сообщения топика в порядке появления в файле; у каждого сообщения есть как минимум `post_id`, `post_index`, `quotes`, `body`, время и автор (поля могут дополняться экспортом, но не обязаны попадать в целевой JSON).

## Политика ошибок

- При **невозможности разобрать файл как корректный YAML** (битый синтаксис, обрыв документа, невалидная структура на этапе чтения multi-doc) обработка **немедленно прекращается** для данного запуска с **кодом ошибки ненулевой** (в CLI) / явным исключением (в библиотечном использовании).
- В отчёте об ошибке обязательно: **путь к файлу**, **тип/текст исключения**, при наличии в данных PyYAML — **позиция (строка/колонка)**, **фрагмент контекста** (например строка или окно символов вокруг места сбоя), при необходимости — указание индекса документа в потоке multi-doc.
- Частично записанный результат при такой ошибке **не считается валидным выходом** (не оставлять «обрезанный» JSON).

## Правила парсинга исходного текста

### Разбиение на сообщения

- Файл читается как **поток YAML-документов** с разделителем `---` между документами (**PyYAML**, режим множественных документов).
- Для построения выходных объектов используются документы с `kind: message`; заголовок топика (`kind: topic`) — источник **общего `title`** для всех постов данного файла и **`topic_id`** для формирования **`id`** поста.

### Поле `text` (основной текст поста — до нормализации итерации 2)

- Источник — строковое поле **`body`** сообщения.
- **`text`** — содержимое `body` **после** служебного блока в начале тела: типично это строка с реферером темы (`Re: …`), имя автора, пустые строки и **строка с датой/временем публикации** сообщения.
- В текущих выгрузках маркер даты оформлен как отдельная строка с ведущим символом **`»`** и датой/временем (не два символа `>>`). При реализации парсера нужно:
  - поддержать фактический шаблон из корпуса;
  - заложить **настраиваемый маркер** (или набор маркеров), чтобы учесть вариант с префиксом **`>>`**, если он появится в других экспортах.
- Всё, что идёт **после** этой строки с датой до конца `body`, попадает в промежуточный **`text`** (итерация 1); далее итерация 2 применяет **нормализацию** к этому полю и к элементам **`quotes`**.

### Поле `quotes` (до и после нормализации)

- Из массива `quotes` сообщения отбираются только элементы с **`depth: 1`**.
- **Для записи в JSON (итерация 1):** из значения поля **`text`** каждого такого элемента **удаляется служебная первая строка** вида **`<автор> писал(а):`**, если она присутствует в начале текста (после нормализации переносов только первая строка сравнивается). Сюда входят конкретные ники из экспорта и **шаблонный префикс** **`%username% писал(а):`**. Удаляется также цепочка **пустых строк** сразу после этой строки до начала собственно цитируемого текста. Остальное содержимое цитаты переносится **без изменений** (ещё не нормализация итерации 2).
- Итерация 2 **заменяет** элементы массива **`quotes`** на нормализованные строки по полным правилам нормализации (пробелы, остатки «писал(а):» внутри текста при необходимости, строка даты **`»` / `>>`**, и т. д.) — тем же правилам, что и для **`text`**.

## Нормализация текста (итерация 2)

Одинаково применяется к строке **`text`** поста и к каждой строке в **`quotes`** (по очереди, независимо).

1. **Пробелы и переносы:** последовательности пробельных символов (включая `\r`, `\n`, табуляцию) сжимаются до **одного пробела**; начало и конец строки **trim**.
2. **Шаблон «автор писал(а):»:** удаляются фрагменты, соответствующие обращению к цитируемому автору в типичной разметке форума, включая варианты вида **`%user% писал(а):`** (плейсхолдер в требовании) и эквиваленты с конкретным ником, например **`lalechka писал(а):`** — по **регулярному выражению**, охватывающему имя/ник (включая пробелы при необходимости) до двоеточия и следующего за ним разделителя/переноса. Цель — оставить собственно цитируемый текст.
3. **Строка даты в теле цитаты:** удаляются строки (или подстроки после нормализации переносов), совпадающие с паттерном **строки даты публикации** в стиле экспорта: ведущий **`»`** или **`>>`**, дата, время (как в примере **`» 03 июн 2025, 11:37`**).
4. Повторить шаг 1 в конце на случай образовавшихся двойных пробелов.

Нормализованные значения с этого шага считаются **каноническими** для дальнейшего **RapidFuzz** (итерация 3) и для выходного JSON.

## Выходной формат: JSON на топик

### Корневой объект

Один UTF-8 JSON-файл на топик. Корень — объект верхнего уровня, удобный для последующего **извлечения отдельных постов** как самостоятельных JSON-объектов.

Рекомендуемая структура:

- **`topic_id`** (string) — из документа `kind: topic`.
- **`title`** (string) — заголовок топика.
- **`posts`** (array) — массив объектов поста **строго в порядке возрастания `post_index`** (каждый элемент — полная схема поста ниже). Извлечение «отдельного json» = взять `posts[i]` без остального файла при необходимости склеить с `topic_id`/`title` из корня.

При необходимости в корень допускается включать прочие поля из `kind: topic` (например `source_url`, `forum_id`), если они нужны downstream — без изменения схемы элементов `posts`.

### Схема одного поста (`posts[]`)

| Поле | Тип | Смысл |
|------|-----|--------|
| `id` | string | `topic_<topic_id>_post_<post_id>` (пример: `topic_2056935_post_2877390`). |
| `post_index` | integer | Порядковый номер поста в топике из поля `post_index` сообщения. |
| `title` | string | Заголовок топика (дублируется в каждом посте для самодостаточности при извлечении). |
| `text` | string | Основной текст поста после **нормализации** (итерация 2). |
| `embedding_text` | null \| string | На этапе структурных объектов — **`null`**; заполнение — следующий этап. |
| `quotes` | array of string | Тексты цитат с `depth: 1` после **нормализации** (итерация 2). |
| `quotes_out` | array of string | Список **`id`** постов **предшествующих** текущему (`post_index` меньше), сопоставленных с цитатами (итерация 3). |
| `quotes_in` | array of string | Обратный индекс по итогам итерации 3 (итерация 4). |

Требования к сериализации:

- Кодировка UTF-8.
- Идентификаторы в `quotes_out` / `quotes_in` — **без дубликатов** в рамках одного поста.
- Порядок в `quotes_out`: по возрастанию **`post_index`** связанных постов (при равных score при линковке — ниже).
- `quotes_out` и `quotes_in` в выгрузке — **пустые массивы** `[]`, если связей нет.

## Пайплайн обработки (четыре итерации)

Обработка ведётся **итерационно** над множеством постов одного топика (и далее — по всем файлам в `data/`).

### Итерация 1 — базовый разбор

Заполняются: **`id`**, **`post_index`**, **`title`**, **`text`** (из `body` по правилам парсинга, **без** нормализации), **`quotes`** (массив строк из YAML для элементов с `depth: 1` **с вырезанием ведущей строки** «`<автор> писал(а):`», включая **`%username% писал(а):`**, см. раздел «Поле `quotes`»), **`embedding_text: null`**, **`quotes_out: []`**, **`quotes_in: []`**.

### Итерация 2 — нормализация `text` и `quotes`

На месте обновляются **`text`** и каждый элемент **`quotes`** по разделу «Нормализация текста». Связи пока не меняются (`quotes_out` / `quotes_in` пустые).

### Итерация 3 — `quotes_out` (RapidFuzz)

Сопоставление выполняется по **нормализованным** строкам: каждая строка в **`quotes`** текущего поста сравнивается с полем **`text`** каждого **предыдущего** поста (`post_index` строго меньше).

**Метрика и порог (зафиксировано для баланса точности и полноты на форумных цитатах):**

- Для пары *(цитата `q`, текст поста `t`)* считаются два скора из **RapidFuzz** (`rapidfuzz.fuzz`): **`partial_ratio(q, t)`** и **`token_sort_ratio(q, t)`**.
- **`partial_ratio`** ловит случай, когда цитата — почти непрерывный фрагмент текста поста (типично для форумов).
- **`token_sort_ratio`** ослабляет влияние перестановки слов и лишних токенов после нормализации.
- Итоговый скор: **`score = max(partial_ratio(q, t), token_sort_ratio(q, t))`**.
- Совпадение **принимается**, если **`score >= 85`**. Порог **85** выбран как разумный компромисс на «живом» тексте после нормализации; при необходимости тонкой настройки на конкретном корпусе допустим **один** вынесенный в конфиг параметр порога с дефолтом **85**, без смены формулы скора.

**Выбор поста:**

- Среди постов-кандидатов с **`post_index` < текущего** выбирается пост с **максимальным** `score`.
- Если **ни у одного** кандидата **`score >= 85`**, для данной цитаты в **`quotes_out` ничего не добавляется**.
- Если несколько кандидатов дают один и тот же максимальный `score`, выбирается пост с **большим `post_index`** (ближайший по контексту среди равных).
- Разные цитаты одного поста могут дать один и тот же **`id`** в **`quotes_out`**; в массиве **`quotes_out`** **`id`** **дедуплицируются**.

### Итерация 4 — `quotes_in`

- Строится **обратный индекс**: для постов A и B, если **`id(A)`** входит в **`quotes_out`** поста B, то **`id(B)`** добавляется в **`quotes_in`** поста A.
- Дубликаты в **`quotes_in`** недопустимы; порядок — по возрастанию **`post_index`** источников.

## Архитектура сервиса (этап структурных объектов)

1. **Reader** — чтение файла топика, разбор YAML multi-doc (**PyYAML**); при ошибке — политика из раздела «Политика ошибок».
2. **Parser (итерация 1)** — извлечение сырого `text` и `quotes`, идентификаторы, пустые связи.
3. **Text normalizer (итерация 2)** — нормализация `text` и `quotes`.
4. **Linker (итерация 3)** — `quotes_out` через **RapidFuzz** по правилам выше.
5. **Reverse index (итерация 4)** — `quotes_in`.
6. **Writer** — один JSON на топик: `topic_id`, `title`, `posts[]` согласно «Выходной формат».

CLI / `uv run` — по мере появления репозитория; интеграция с Qdrant и эмбеддером — следующие версии vision.
