# План разработки по этапам

Технические детали формата данных, парсинга и JSON — в [vision.md](vision.md). Ниже только **этапы**: каждый завершается **проверяемым** инкрементом (команда, артефакт, сценарий).

Этапы **1–5** закрывают **итерацию 1** пайплайна и вывод JSON с сырыми (до нормализации) `text` / `quotes` в смысле vision — фактически после итерации 1; дальше по [vision.md](vision.md) — **итерации 2–4** (нормализация, связи по цитатам).

---

## Этап 1 — репозиторий и игнорирование лишних файлов

**Цель:** проект готов к публикации на GitHub и локальной работе без мусора в статусе git.

**Состав работ:**

- Добавить `.gitignore`: виртуальное окружение (например `.venv/`), кэши Python (`__pycache__/`, `*.pyc`), артефакты IDE, секреты (`.env`, ключи), при необходимости каталог выхода (`out/` или согласованное имя).
- Инициализировать git-репозиторий (если ещё нет), первый осмысленный коммит с правилами из принятых в проекте договорённостей.
- Подготовить удалённый репозиторий GitHub и привязать `origin` (инструкция или выполнение вручную — по договорённости).

**Проверка:** `git status` чистый после типичных команд разработки; удалённый репозиторий создан и доступен.

---

## Этап 2 — зависимости и каркас Python-проекта

**Цель:** воспроизводимое окружение и точка входа для кода.

**Состав работ:**

- Добавить `pyproject.toml` с пакетом/приложением (minimal).
- Зафиксировать зависимости через **uv**: как минимум **PyYAML** (разбор multi-doc, см. vision). **RapidFuzz** — подключить в этом же этапе как зависимость проекта (использование — на последующих этапах пайплайна), чтобы не дробить инфраструктуру.
- Сгенерировать и закоммитить **`uv.lock`**.
- Минимальный **README**: как установить зависимости (`uv sync`) и как в дальнейшем запускать CLI (заполняется по мере этапов 3–5).

**Проверка:** `uv sync` выполняется без ошибок; зависимости импортируются в интерпретаторе.

---

## Этап 3 — чтение файла топика и политика ошибок YAML

**Цель:** надёжно прочитать `data/topic_<id>.txt` как поток YAML-документов и валидно отработать сбои.

**Состав работ:**

- Реализовать загрузку multi-doc через PyYAML.
- Извлечь первый документ `kind: topic` (минимум `topic_id`, `title` для следующих этапов); распознать последующие `kind: message`.
- При **битом YAML** — **немедленное прекращение** с **подробным отчётом**: путь к файлу, тип/текст ошибки, при доступности у PyYAML — позиция, контекст, при необходимости номер документа в потоке (как в vision).

**Проверка:** успешный прогон на одном реальном файле из `data/`; искусственно повреждённый файл даёт ожидаемый отчёт и ненулевой код выхода (CLI) или исключение (библиотечный вызов).

---

## Этап 4 — итерация 1: поля поста (разбор `body` и `quotes`)

**Цель:** из каждого `kind: message` получить структуру поста согласно **итерации 1** vision.

**Состав работ:**

- Сформировать `id`: `topic_<topic_id>_post_<post_id>`.
- Проставить `post_index`, дублировать в посте `title` топика.
- Выделить **`text`** из `body`: всё после строки с датой публикации (маркеры `»` / при необходимости `>>`, см. vision).
- Собрать **`quotes`**: только элементы с `depth: 1`, значения поля `text` **с вырезанием служебной первой строки «… писал(а):»** (включая шаблон `%username% писал(а):`), остальное без нормализации итерации 2.
- Установить **`embedding_text`: `null`**, **`quotes_out`: `[]`**, **`quotes_in`: `[]`**.

**Проверка:** для выбранного топика сравнить вручную или скриптом несколько постов с исходным `.txt` (границы `text`, список цитат).

---

## Этап 5 — вывод JSON на топик и CLI

**Цель:** один файл результата на топик, удобный для вырезания отдельных постов.

**Состав работ:**

- Собрать корневой объект: `topic_id`, `title`, **`posts`** — массив постов по возрастанию `post_index` (схема полей — vision).
- Запись в UTF-8 (путь: один файл на входной топик, например `out/topic_<id>.json` — имя зафиксировать в README).
- CLI (или эквивалент `uv run …`): аргументы — входной файл топика и выходной путь (или каталог назначения).

**Проверка:** `uv run …` на файле из `data/` создаёт валидный JSON; `posts[i]` можно использовать как отдельный объект поста.

---

## Этап 6 — итерация 2: модуль нормализации текста

**Цель:** одна чистая функция (или небольшой модуль), реализующая раздел **«Нормализация текста (итерация 2)»** в vision, применимая **к одной строке** так же для **`text` поста**, так и для **каждого элемента `quotes`**.

**Состав работ:**

- Шаг **1** (vision): схлопывание пробельных символов (включая `\r`, `\n`, таб) до **одного пробела**, **trim** начала/конца.
- Шаг **2**: удаление фрагментов «`<автор> писал(а):`» (включая **`%user%` / `%username%`** и ники с пробелами), по **регулярному выражению**, согласованному с vision.
- Шаг **3**: удаление строк/фрагментов, соответствующих **строке даты** в стиле экспорта: ведущий **`»`** или **`>>`**, дата и время.
- Шаг **4**: повторить шаг 1 в конце.

**Проверка:** на коротких фикстурах и/или на фрагментах из реальных топиков вручную или скриптом видно совпадение с ожиданиями vision; при необходимости добавить минимальные вызовы из `uv run python -c "..."`.

---

## Этап 7 — итерация 2: встраивание в пайплайн и JSON

**Цель:** экспортируемый топик в JSON содержит **`text`** и **`quotes`**, прошедшие нормализацию итерации 2; **`quotes_out`** / **`quotes_in`** по-прежнему **`[]`**; **`embedding_text`** — **`null`**.

**Состав работ:**

- После построения постов итерации 1 применить нормализацию **на месте** к каждому `text` и к каждой строке в `quotes` (порядок постов и длина массива `quotes` не меняются).
- Подключить шаг к существующему пути **CLI → JSON** (или явный флаг вида `--pipeline 2` — по согласованию в коде; по умолчанию разумно выставить **итерацию 2 как текущий целевой экспорт**, если не нужна обратная совместимость с «сырым» JSON).
- Обновить **README**: что в выходном файле поля `text` / `quotes` уже нормализованы.

**Проверка:** `uv run forum-etl …` даёт JSON, где для известного топика нет лишних переносов/префикса даты внутри нормализованных строк при соответствующих входах; парсинг не ломается.

---

## Этап 8 — итерация 3: `quotes_out` (RapidFuzz)

**Цель:** для каждой нормализованной цитаты — не более одного `id` предыдущего поста по правилам vision (порог **85**, `max(partial_ratio, token_sort_ratio)`), дедупликация в `quotes_out`.

**Состав работ:**

- Реализация линковки по **`post_index`**; кандидаты только с меньшим индексом.
- Конфигурируемый при необходимости **порог** (дефолт из vision).

**Проверка:** на выбранном топике доли осмысленных связей проверяются выборочно; нет исключений на типовых файлах из `data/`.

---

## Этап 9 — итерация 4: `quotes_in`

**Цель:** обратный индекс по итогам итерации 3; порядок и отсутствие дубликатов — по vision.

**Состав работ:**

- Однопроходное или двухфазное построение `quotes_in` из финальных `quotes_out`.

**Проверка:** для каждой пары постов консистентность «A в `quotes_out` у B» ⇔ «B в `quotes_in` у A» на выборке.
